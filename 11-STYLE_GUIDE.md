# Style Guide & Tooling

> **Goal:** Enforce the [Google Developer Documentation Style Guide](https://developers.google.com/style) with Vale, catch formatting issues with markdownlint, and stop arguing about commas in PR reviews. Every rule here is backed by a published source, if you think one is wrong, check the [references](#14-sources-and-references) before opening an issue.

---

## 1. Voice and Tone

### 1.1 Active voice

Say who does what. Active voice is shorter, clearer, and doesn't make your reader do mental gymnastics to figure out who's responsible.

* **No:** "The file is generated by the script."
* **Yes:** "The script generates the file."

Passive voice is fine when you're deliberately hiding the actor — error messages ("the connection was refused"), de-emphasizing blame, or when the actor genuinely doesn't matter. Don't use it out of habit.

> Google's [Technical Writing One](https://developers.google.com/tech-writing/one/active-voice) puts it well: "Most readers mentally convert passive voice to active voice. Why subject your readers to extra processing time?" This isn't just Google's opinion — [Microsoft](https://learn.microsoft.com/en-us/style-guide/grammar/verbs), the [Federal Plain Language Guidelines](https://digital.gov/guides/plain-language/writing/), the [UK GDS](https://www.gov.uk/guidance/content-design/writing-for-gov-uk), and the [European Commission](https://translation.ec.europa.eu/languages-and-translation-european-commission/plain-language-making-european-commission-texts-clear_en) all agree. The US [Plain Writing Act of 2010](https://www.congress.gov/bill/111th-congress/house-bill/946) literally made it law for federal agencies.

### 1.2 Present tense

Write about what things *do*, not what they *will* do. Present tense for general behavior, future tense only when something genuinely happens later.

* **General behavior:** "The system connects to the database."
* **Genuinely future:** "The file will be archived the next time the backup runs."

Don't write "the server will return a 200" when it already does. That's present tense.

> [Google — Tense](https://developers.google.com/style/tense) allows future tense "when distinguishing actions that occur in the future." The blanket "never use will" advice you see elsewhere is an oversimplification.

### 1.3 Second person

Talk to the reader. Use "you." It's direct, it's clear, and it makes obvious who needs to do what.

* **No:** "The user should configure the environment variable."
* **Yes:** "Configure the environment variable." or "You can configure..."

Every major style guide agrees on this one — [Google](https://developers.google.com/style/person), [Microsoft](https://learn.microsoft.com/en-us/style-guide/grammar/person), [UK GDS](https://www.gov.uk/guidance/content-design/writing-for-gov-uk), [Federal Plain Language](https://digital.gov/guides/plain-language/writing/). NNg's research found that [direct, simplified content](https://www.nngroup.com/articles/writing-for-lower-literacy-users/) improved task success rates from 46% to 82%. This isn't style preference — it's measurably better.

### 1.4 Tone matrix

Different docs need different tones. A runbook during an incident is not the place for encouraging language, and an API reference is not the place for personality.

This draws from the [Diataxis framework](https://diataxis.fr/) (which nails the "different docs serve different purposes" concept) and [NNg's tone research](https://www.nngroup.com/articles/tone-voice-users/) (which proved tone measurably affects trust — 52% of desirability scores came from trustworthiness).

| Document Type | Tone | Example | Why |
|---------------|------|---------|-----|
| Tutorial / Getting Started | Encouraging, collaborative | "You're ready to deploy." | Diataxis: ["The first-person plural affirms the relationship between tutor and learner."](https://diataxis.fr/tutorials/) |
| How-To Guide | Practical, imperative | "Run the migration script." | Diataxis: ["Action and only action"](https://diataxis.fr/how-to-guides/) — like a recipe. |
| API Reference | Neutral, precise | "Returns the resource or `404`." | Diataxis: ["Austere and uncompromising... neutrality, objectivity, factuality."](https://diataxis.fr/reference/) |
| Architecture / ADR | Analytical, discursive | "We chose X because Y." | Diataxis: ["Admit opinion and perspective."](https://diataxis.fr/explanation/) |
| Troubleshooting | Empathetic, direct | "This error occurs when..." | [Google](https://developers.google.com/style/tone): Sound like "a knowledgeable friend." |
| Runbook / Incident | Urgent, imperative | "Restart the service immediately." | Under pressure, clarity saves time. |
| CHANGELOG | Factual, brief | "Add retry logic to auth flow." | [Keep a Changelog](https://keepachangelog.com/): scannable, no fluff. |

## 2. Inclusive Language

This matters. Not because it's trendy — because exclusionary language alienates contributors and the replacements are genuinely better descriptions anyway. "Allowlist" is more descriptive than "whitelist." "Primary/replica" tells you what the relationship actually is.

### Replacement Reference

| Avoid | Use Instead | Why / Source |
|-------|-------------|--------------|
| whitelist / blacklist | allowlist / denylist | [Inclusive Naming Initiative (Tier 1)](https://inclusivenaming.org/word-lists/tier-1/whitelist/), [IETF draft-knodel-terminology](https://datatracker.ietf.org/doc/html/draft-knodel-terminology-10), [Google](https://developers.google.com/style/inclusive-documentation) |
| master / slave | primary / replica, leader / follower | [IETF](https://datatracker.ietf.org/doc/html/draft-knodel-terminology-10), [Django](https://code.djangoproject.com/ticket/22667), [Redis](https://redis.io/docs/latest/commands/replicaof/), [GitHub](https://github.blog/changelog/2020-10-01-the-default-branch-for-newly-created-repositories-is-now-main/) |
| sanity check | confidence check, validation check | [Inclusive Naming (Tier 2)](https://inclusivenaming.org/word-lists/tier-2/sanity-check/), [Google Word List](https://developers.google.com/style/word-list) |
| dummy value | placeholder value | [Google Word List](https://developers.google.com/style/word-list): "Don't use 'dummy.' Use 'placeholder.'" |
| he/she | they, the user | [Google](https://developers.google.com/style/pronouns), [Microsoft](https://learn.microsoft.com/en-us/style-guide/bias-free-communication) |
| kill the process | stop, end, terminate | [Google Word List](https://developers.google.com/style/word-list) |
| crippled | degraded, limited | [Google](https://developers.google.com/style/inclusive-documentation), [Inclusive Naming Initiative](https://inclusivenaming.org/) |
| grandfathered | legacy, exempt | [Inclusive Naming (Tier 1)](https://inclusivenaming.org/word-lists/tier-1/grandfathered/) |

This isn't a comprehensive list — [Microsoft](https://learn.microsoft.com/en-us/style-guide/bias-free-communication), [Apple](https://support.apple.com/guide/applestyleguide/writing-inclusively-apsg1a3a0436/web), and the [ACM](https://www.acm.org/diversity-inclusion/words-matter) all publish their own guidelines. The point is: when you see a term that has an inclusive alternative and the alternative is equally clear, use it.

## 3. Formatting and Structure

### 3.1 Headings — sentence case

Capitalize the first word and proper nouns. That's it. Don't Title Case Your Headings Like A Book Cover.

* **No:** "Getting Started With The API"
* **Yes:** "Getting started with the API"

[Google](https://developers.google.com/style/headings), [Microsoft](https://learn.microsoft.com/en-us/style-guide/capitalization) ("When in doubt, don't capitalize"), [GitLab](https://docs.gitlab.com/development/documentation/styleguide/), [UK GDS](https://www.gov.uk/guidance/content-design/writing-for-gov-uk) — everyone agrees on this. There is no style guide that recommends title case for documentation headings.

### 3.2 Oxford comma

Use it. Always. The serial comma before "and" or "or" in a list of three or more.

* **No:** "HTML, CSS and JavaScript"
* **Yes:** "HTML, CSS, and JavaScript"

[Google](https://developers.google.com/style/commas), [Microsoft](https://learn.microsoft.com/en-us/style-guide/top-10-tips-style-voice), [Apple](https://support.apple.com/guide/applestyleguide/intro-apsgb744e4a3/web), and [GitLab](https://docs.gitlab.com/development/documentation/styleguide/) all require it. GitLab even enforces it with Vale. This is one of those rare cases where the industry has reached consensus — take the win and move on.

### 3.3 Lists

Numbered lists for steps (order matters). Bullets for everything else (order doesn't matter). You can nest bullets inside numbered steps — that's fine.

> [Google](https://developers.google.com/style/lists) and [Microsoft](https://learn.microsoft.com/en-us/style-guide/scannable-content/lists) both say the same thing here. Microsoft adds a useful guideline: "no more than seven items" in a list when possible.

### 3.4 Contractions

Use them. "Don't" is actually *clearer* than "do not" — it's harder to accidentally misread "don't" as "do" than to miss the "not" in "do not."

Avoid weird ones: "guides're", "mightn't've", or noun+verb ("Microsoft's developing"). Stick to the common ones: don't, can't, you're, it's, there's.

> [Google](https://developers.google.com/style/contractions) specifically recommends negation contractions for this reason. [Microsoft](https://learn.microsoft.com/en-us/style-guide/word-choice/use-contractions) agrees. One exception: the [UK GDS](https://www.gov.uk/guidance/content-design/writing-for-gov-uk) avoids negative contractions ("cannot" over "can't") — if you're writing for government audiences, follow their lead.

### 3.5 Em dashes

Use spaces around em dashes. It's easier to read, especially in monospace fonts and on narrow screens.

* **No:** "Vale catches problems—before they reach production."
* **Yes:** "Vale catches problems — before they reach production."

The unspaced style is traditional in print typography (where proportional fonts give the dash enough breathing room). In code repos, terminals, and Markdown renderers — where everything is monospace or variable-width — the spaced version is clearer. [Microsoft](https://learn.microsoft.com/en-us/style-guide/punctuation/dashes-hyphens/em-dashes) uses spaced em dashes. The [UK GDS](https://www.gov.uk/guidance/content-design/writing-for-gov-uk) uses spaced en dashes for the same reason. Google's style guide doesn't take a position, so we side with readability.

### 3.6 Abbreviations

Spell it out on first use, abbreviation in parentheses. Then use the abbreviation going forward.

* **First time:** "Architecture Decision Record (ADR)"
* **After that:** "ADR"

You don't need to spell out API, URL, HTML, or CSS. Everyone knows those. Use your judgement — if your audience knows the acronym, skip the ceremony.

> [Google](https://developers.google.com/style/abbreviations) and [Microsoft](https://learn.microsoft.com/en-us/style-guide/acronyms) both document this convention.

### 3.7 UI interaction verbs

"Click" assumes a mouse. Not everyone uses one. Use device-agnostic verbs instead.

| Action | Use | Don't Use | Source |
|--------|-----|-----------|--------|
| Button, link, menu item | **Select** | Click | [Microsoft](https://learn.microsoft.com/en-us/style-guide/procedures-instructions/describing-interactions-with-ui) |
| Checkbox, radio button | **Select** / **Clear** | Click, Check | [Google](https://developers.google.com/style/ui-elements) |
| Text field | **Enter** | Type in | [Microsoft](https://learn.microsoft.com/en-us/style-guide/procedures-instructions/describing-interactions-with-ui) |
| Menu command | **Choose** | Click on | [Apple](https://support.apple.com/guide/applestyleguide/c-apsgb744e4a3/web) |
| Toggle | **Turn on** / **Turn off** | Click | [Microsoft](https://learn.microsoft.com/en-us/style-guide/a-z-word-list-term-collections/t/turn-on-turn-off) |

Apple retains "click" for macOS-specific docs, and that's fine — platform-specific content can use platform-specific verbs. For general documentation, "select" is the safer default.

## 4. Google style highlights

Non-obvious rules from the [Google Developer Documentation Style Guide](https://developers.google.com/style) that trip people up:

* **Don't document the future.** If the feature doesn't exist yet, it doesn't go in the docs — unless legal has approved a "Beta" or "Preview" banner. Source: [Google — Future Features](https://developers.google.com/style/future).
* **Conditions come first.** "If you're ready, select Submit" — not "Select Submit if you're ready." The reader shouldn't act before they've read the condition. Source: [Google — Highlights](https://developers.google.com/style/highlights).
* **Drop "should."** It's ambiguous. Is it required or optional? Use "must" for requirements, or rephrase as imperative. Source: [Google — Prescriptive Documentation](https://developers.google.com/style/prescriptive-documentation).
* **Stop minimizing.** "Simply run the command" implies the reader is stupid if they find it difficult. Cut "simply", "just", "easy", "obviously", and "straightforward." Source: [Google](https://developers.google.com/style/word-list), [Microsoft](https://learn.microsoft.com/en-us/style-guide/top-10-tips-style-voice), [Write the Docs Prague 2018](https://www.writethedocs.org/videos/prague/2018/don-t-say-simply-jim-fisher/), [justsimply.dev](https://justsimply.dev/).
* **Skip "please."** Instructions are not requests. "Run the command" — not "Please run the command." Use "please" only when you're asking for forgiveness or genuinely inconveniencing the reader. Source: [Google](https://developers.google.com/style/word-list), [Microsoft](https://learn.microsoft.com/en-us/style-guide/a-z-word-list-term-collections/p/please).

## 5. Terminology consistency

Inconsistent terminology confuses people. If you call it "PostgreSQL" in one doc and "Postgres" in another, readers waste time wondering if they're different things. [Jakob Nielsen](https://www.nngroup.com/articles/ten-usability-heuristics/) made "Consistency and Standards" his fourth usability heuristic back in 1994, and it's still the one most documentation violates.

Two things to get right:

* **Capitalization:** "URL" not "Url", "ID" not "Id", "GitHub" not "Github."
* **Project vocabulary:** Define canonical terms in `styles/config/vocabularies/` so Vale catches variations automatically.

### Creating a project vocabulary

```text
styles/config/vocabularies/Base/
├── accept.txt    # Correct spellings (won't be flagged)
└── reject.txt    # Wrong spellings (flagged as errors)
```

Entries are case-sensitive regex, one per line. `#` for comments.

**accept.txt:**

```text
Kubernetes
GitHub
PostgreSQL
frontmatter
```

**reject.txt:**

```text
Github
Kubernete
Postgre
```

Set `Vocab = Base` in `.vale.ini`. Details: [Vale — Vocabulary](https://vale.sh/docs/keys/vocab).

## 6. Code sample conventions

Keep code comments consistent across your docs. These conventions are drawn from established practices — some are formal standards, some are community conventions that work well enough that everyone uses them.

| Comment Type | Convention | Example | Where It Comes From |
|--------------|-----------|---------|---------------------|
| Explanation | `#` prefix, sentence case | `# Connect to the database` | Universal |
| Output | `# =>` prefix | `# => {"status": "ok"}` | Ruby community convention — used throughout the [official docs](https://docs.ruby-lang.org/en/3.3/) |
| Placeholder | `<ANGLE_BRACKETS>` | `export API_KEY=<YOUR_API_KEY>` | [POSIX](https://pubs.opengroup.org/onlinepubs/9699919799/basedefs/V1_chap12.html), [Microsoft](https://learn.microsoft.com/en-us/style-guide/developer-content/formatting-developer-text-elements) |
| Omission | Descriptive comment | `# ... (remaining config)` | [Google](https://developers.google.com/style/code-samples), [GitLab](https://docs.gitlab.com/development/documentation/styleguide/) |
| Warning | `# WARNING:` prefix | `# WARNING: This deletes all data` | Common convention (no formal standard) |

Worth knowing: [Google](https://developers.google.com/style/placeholders) uses UPPER_SNAKE_CASE without angle brackets for placeholders, and [GitHub](https://docs.github.com/en/contributing/style-guide-and-content-model/style-guide) uses ALL-CAPS-KEBAB-CASE. We went with angle brackets because they're the oldest convention (POSIX, 1988) and the most instantly recognizable.

## 7. Units and terminology

### 7.1 Binary vs decimal units

This has been the standard since [IEC 60027-2:1999](https://physics.nist.gov/cuu/Units/binary.html), endorsed by [NIST](https://physics.nist.gov/cuu/Units/binary.html) and [IEEE 1541-2002](https://standards.ieee.org/ieee/1541/4880/). Not new — just underadopted.

| Unit | Base | Value | Use For |
|------|------|-------|---------|
| **GiB** (gibibyte) | 1024 | 2^30 = 1,073,741,824 bytes | RAM, disk partitions, binary contexts |
| **GB** (gigabyte) | 1000 | 10^9 = 1,000,000,000 bytes | Network bandwidth, SI contexts |

If you write "8GB of RAM" you're technically wrong — RAM is measured in powers of 2. It's "8 GiB." Most people won't notice, but the ones who do are the ones reviewing your architecture docs.

### 7.2 Hyphenation

These catch people out. Both Google and Microsoft agree on the rules, they're just not intuitive.

| Term | Correct Form | Source |
|------|-------------|--------|
| real-time (adjective) | "real-time analytics" | [Microsoft](https://learn.microsoft.com/en-us/style-guide/a-z-word-list-term-collections/r/real-time), [Google](https://developers.google.com/style/word-list) |
| real time (noun) | "processed in real time" | [Microsoft](https://learn.microsoft.com/en-us/style-guide/a-z-word-list-term-collections/r/real-time) |
| anti-malware (adjective) | "anti-malware tools" | [Microsoft](https://learn.microsoft.com/en-us/style-guide/a-z-word-list-term-collections/a/anti-malware) |
| antivirus (closed) | "antivirus software" | Microsoft (inconsistent with anti-malware, but that's what they specify) |

## 8. Common mistakes

Every one of these shows up in real PRs. Every one has a published source explaining why it's wrong.

| Mistake | The Problem | Fix | Source |
|---------|------------|-----|--------|
| "Click here" | Screen readers list links by text — "click here" means nothing out of context | Descriptive link text: "See the [API docs](...)" | [WCAG 2.4.4](https://www.w3.org/WAI/WCAG21/Understanding/link-purpose-in-context.html), [Section508.gov](https://www.section508.gov/blog/accessibility-bytes/descriptive-links-and-hypertext/) |
| Bold text as headings | Screen readers can't navigate it — it's just bold, not structure | Use `##` heading levels | [WCAG 1.3.1](https://www.w3.org/WAI/WCAG21/Understanding/info-and-relationships.html), [W3C H42](https://www.w3.org/WAI/GL/WCAG20/WD-WCAG20-TECHS-20080505/H42.html) |
| Images without alt text | Invisible to screen reader users | Add descriptive alt text | [WCAG 1.1.1](https://www.w3.org/WAI/WCAG22/Understanding/non-text-content.html), [W3C F65](https://www.w3.org/WAI/WCAG21/Techniques/failures/F65) |
| "Simply run..." | Dismissive — alienates anyone who finds it difficult | "Run..." | [Google](https://developers.google.com/style/word-list), [Microsoft](https://learn.microsoft.com/en-us/style-guide/top-10-tips-style-voice), [justsimply.dev](https://justsimply.dev/) |
| "Obviously..." | Shames readers who don't find it obvious | Remove it, or explain | [justsimply.dev](https://justsimply.dev/) |
| Conditions after instructions | "Select Submit if you're ready" — reader acts before reading the condition | Conditions first: "If you're ready, select Submit" | [Google](https://developers.google.com/style/highlights) |
| Undefined acronyms | Reader hits "ADR" with no idea what it means | Spell out on first use | [Google](https://developers.google.com/style/abbreviations) |
| Jargon without context | Excludes anyone outside your domain | Define it or link to a glossary | [Google — Jargon](https://developers.google.com/style/jargon) |
| "&" instead of "and" | Informal, inconsistent | Spell out "and" | [Google](https://developers.google.com/style/highlights) |
| "Please" in procedures | Instructions aren't requests | Imperative: "Run the command" | [Google](https://developers.google.com/style/word-list), [Microsoft](https://learn.microsoft.com/en-us/style-guide/a-z-word-list-term-collections/p/please) |

## 9. Readability

Your docs don't need to read like a textbook. [NNg's research](https://www.nngroup.com/articles/writing-for-lower-literacy-users/) showed that rewriting content at a lower reading level improved task success for *everyone* — not just lower-literacy users. Higher-literacy users went from 68% to 93% success. The quote that should end every readability debate: "No one has ever complained that a text was too easy to understand."

### Target reading levels

| Audience | Target | Source |
|----------|--------|--------|
| General / landing pages | Grade 6-8 | [NNg](https://www.nngroup.com/articles/plain-language-experts/) |
| Developer docs | Grade 8-10 | Industry practice |
| Expert / internal | Grade 10-12 | [NNg](https://www.nngroup.com/articles/plain-language-experts/) |

### Where the metrics come from

| Metric | Original Source | What It Does |
|--------|----------------|--------------|
| **Flesch Reading Ease** | Flesch, *J. Applied Psychology* 32(3), 1948 | Score 0-100 from sentence length + syllable count |
| **Flesch-Kincaid** | Kincaid et al., Naval Technical Training Command, 1975 | Converts Flesch to US grade level |
| **Gunning Fog** | Gunning, *The Technique of Clear Writing*, McGraw-Hill, 1952 | Years of education needed |
| **Coleman-Liau** | Coleman & Liau, *J. Applied Psychology* 60(2), 1975 | Grade level from character count (no syllable counting) |

Vale's [Readability package](https://github.com/errata-ai/Readability) runs all of these automatically. They're set to `suggestion` level in our config — they'll tell you when something is hard to read, but they won't block your commit.

## 10. Automated enforcement

### Vale (prose linting)

[Vale](https://vale.sh/) is an open-source prose linter that actually understands Markdown structure — it won't flag code blocks, frontmatter, or URLs. It's used in production by [GitLab](https://docs.gitlab.com/development/documentation/testing/vale/), [Grafana](https://grafana.com/docs/writers-toolkit/review/lint-prose/), [Datadog](https://www.datadoghq.com/blog/engineering/how-we-use-vale-to-improve-our-documentation-editing-process/), [Crossplane](https://docs.crossplane.io/contribute/vale/), and [Red Hat](https://github.com/redhat-documentation/vale-at-red-hat).

#### Style packages

We load five packages. Each does one thing well:

| Package | What It Catches | Repo |
|---------|----------------|------|
| **Google** | Active voice, tense, person, headings, word list violations | [errata-ai/Google](https://github.com/errata-ai/Google) (community implementation, not officially endorsed by Google) |
| **write-good** | Passive voice, weasel words, wordy phrases | [errata-ai/write-good](https://github.com/errata-ai/write-good) |
| **proselint** | Redundancy, jargon, illogic, cliches | [errata-ai/proselint](https://github.com/errata-ai/proselint) |
| **alex** | Insensitive or inconsiderate language | [errata-ai/alex](https://github.com/errata-ai/alex) |
| **Readability** | Flesch-Kincaid, Gunning Fog, Coleman-Liau, SMOG, LIX, ARI | [errata-ai/Readability](https://github.com/errata-ai/Readability) |

**Why not Microsoft?** Too much overlap with Google — both flag first-person, passive voice, and similar patterns. You end up with duplicate alerts. [Crossplane's Vale guide](https://docs.crossplane.io/contribute/vale/) documents this exact problem. We use Google as the base and fill gaps with write-good, proselint, and alex.

#### Rules we disable (and why)

| Rule | Why It's Off |
|------|-------------|
| `Google.Latin` | e.g. and i.e. are fine in technical writing |
| `Google.We` | "We recommend..." is normal in project docs |
| `Google.Will` | Future tense is fine for genuinely future actions |
| `write-good.E-Prime` | Flags every "is", "are", "was", "were." Absolutely unusable — almost every project disables this |
| `write-good.TooWordy` | Flags phrases that are sometimes the right choice |
| `write-good.So` | "So" is a valid sentence starter in conversational docs |

Readability metrics are all set to `suggestion` — they inform, they don't block.

#### Alert levels

| Where | Level | Why |
|-------|-------|-----|
| Your editor | `suggestion` | See everything. [GitLab](https://docs.gitlab.com/development/documentation/testing/vale/) and [Grafana](https://grafana.com/docs/writers-toolkit/review/lint-prose/) both use this locally. |
| CI pipeline | `error` | Only block on real problems. Override with `--minAlertLevel=error`. |
| Pre-commit | `error` | Don't slow people down with suggestions on every commit. |

#### Setup

The `init.sh` installer copies everything you need into your repo:

* **`.vale.ini`** — Vale configuration (style packages, disabled rules, alert levels)
* **`styles/config/vocabularies/Base/`** — project vocabulary (`accept.txt` and `reject.txt`) so Vale knows your project's correct spellings
* **`.pre-commit-config.yaml`** — pre-commit hooks that run Vale and markdownlint on every commit

Once installed, Vale runs automatically on every commit via pre-commit — no manual steps required.

```bash
# Via pre-commit (runs automatically on every commit)
git commit

# Manually (for one-off linting or CI)
brew install vale        # macOS
choco install vale       # Windows
vale sync                # Download style packages
vale .                   # Lint the entire repo
```

#### Config

The `.vale.ini` at repo root covers **all** Markdown files in the repository — not just a `docs/` folder. Run `vale .` from the repo root to lint everything.

```ini
StylesPath = styles
MinAlertLevel = suggestion
Packages = Google, write-good, proselint, alex, Readability
Vocab = Base

# ── Markdown files ──────────────────────────────────────────────
[*.md]
BasedOnStyles = Vale, Google, write-good, proselint, alex, Readability

# Google — disable overly aggressive or context-dependent rules
Google.Latin = NO
Google.We = NO
Google.Will = NO

# write-good — disable rules that produce excessive noise
write-good.E-Prime = NO
write-good.TooWordy = NO
write-good.So = NO

# Readability — keep as suggestions only
Readability.FleschKincaid = suggestion
Readability.GunningFog = suggestion
Readability.LIX = suggestion
Readability.SMOG = suggestion
Readability.ColemanLiau = suggestion
Readability.AutomatedReadability = suggestion

# ── Plain text files ────────────────────────────────────────────
[*.txt]
BasedOnStyles = Vale, write-good, proselint
```

#### Suppressing rules inline

When you need to break a rule intentionally, use Vale comments:

```markdown
<!-- vale Google.Passive = NO -->
This section was written in passive voice intentionally.
<!-- vale Google.Passive = YES -->
```

Prefer this over global disabling. [Grafana](https://grafana.com/docs/writers-toolkit/review/lint-prose/) does the same.

### Markdownlint (format linting)

[markdownlint-cli2](https://github.com/DavidAnson/markdownlint-cli2) handles formatting: heading levels, list consistency, trailing whitespace, code fence syntax. It runs alongside Vale in [pre-commit](#pre-commit-config) on every commit, auto-fixing what it can.

The `init.sh` installer copies `.markdownlint-cli2.yaml` into your repo with our default rule configuration. Config lives in `.markdownlint-cli2.yaml`.

**Disabled rules:**

| Rule | Why |
|------|-----|
| MD001 | Allow skipping heading levels when needed |
| MD003 | Allow both ATX (`###`) and setext (`===`) heading styles |
| MD013 | Line length limits don't work with URLs and tables |
| MD024 | Same headings in different sections is valid (e.g., "Overview" repeated) |
| MD025 | Multiple top-level headings are common in compliance docs |
| MD028 | Blank lines inside blockquotes are sometimes needed for readability |
| MD033 | Inline HTML is sometimes necessary (`<details>`, `<summary>`, complex formatting) |
| MD036 | Too many false positives on bold terms and definitions — bold-as-heading is still a mistake (see [section 8](#8-common-mistakes)), catch it in review |
| MD040 | Not every code block has a language (plain text output, for example) |
| MD051 | Link fragment validation has false positives with certain doc structures |
| MD056 | Tables sometimes have intentional column variations |
| MD059 | Too many false positives on short but valid link text — "click here" is still a mistake (see [section 8](#8-common-mistakes)), catch it in review |
| MD060 | Too strict for practical use with various table formats |

```bash
# Via pre-commit (runs automatically)
git commit

# Manually
npx markdownlint-cli2 --fix "**/*.md"
```

### Pre-commit config

Both Vale and markdownlint run automatically on every commit via [pre-commit](https://pre-commit.com/). Vale runs at `error` level in pre-commit so it only blocks on real problems — suggestions and warnings show up in your editor but don't slow down commits. Markdownlint runs with `--fix` to auto-correct formatting issues on the fly.

```yaml
- repo: https://github.com/errata-ai/vale
  rev: v3.9.6
  hooks:
    - id: vale
      args: [--minAlertLevel=error]
      types: [markdown]

- repo: https://github.com/DavidAnson/markdownlint-cli2
  rev: v0.20.0
  hooks:
    - id: markdownlint-cli2
      args: [--fix]
```

## 11. Style guide alignment

This standard follows the [Google Developer Documentation Style Guide](https://developers.google.com/style) as its base, supplemented by other published guides. Where they disagree, here's what we chose and why:

| Topic | Our Choice | Agrees With | Disagrees With |
|-------|-----------|-------------|----------------|
| Active voice | Preferred, with exceptions | Google, Microsoft, GDS, Federal Plain Language | — |
| Second person | Default | Google, Microsoft, GDS, Kubernetes | — |
| Contractions | Yes, including negation | Google, Microsoft, Apple | UK GDS (avoids "can't", "don't") |
| Oxford comma | Required | Google, Microsoft, Apple, GitLab | — |
| Headings | Sentence case | Google, Microsoft, Apple, GitLab, GDS | — |
| UI verbs | "Select" as default | Microsoft, Google, GitLab, GDS | Apple (retains "click" for macOS) |
| Placeholders | `<ANGLE_BRACKETS>` | Microsoft, POSIX, GitLab | Google (UPPER_SNAKE), GitHub (CAPS-KEBAB) |
| Em dashes | Spaced (`word — word`) | Microsoft, UK GDS (spaced en dash) | Google (no position), AP/Chicago (unspaced) |
